# Retro 2.0 — "The Watcher"

A persistent brain that makes coding agents better automatically, session over session.

## Problem Statement

Retro 1.x works but doesn't get used. The five-command pipeline (ingest → analyze → apply → review → sync) requires the user to remember to run it. Even with a post-commit hook automating ingestion, the review step is manual, the terminal nudge is easy to miss, and across multiple projects it becomes a chore that gets skipped — defeating the entire purpose.

The core analysis engine is solid. What's broken is the runtime model (manual commands) and the surface (CLI output you have to go look at).

## Vision

- **Zero-effort improvement** — No commands to remember. The coding agent gets better session over session.
- **Noticeable difference** — Yesterday's mistakes don't happen today. The improvement is tangible.
- **Suggestions come to you** — Improvements surface themselves in-session and via a dashboard, ready to approve with minimal friction.
- **You stay in control** — Visibility into what's being learned and applied. Configurable trust level.
- **It's automatic** — The full loop from "observe mistake" to "suggest fix" to "easy approval" happens without user initiation.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture approach | "The Watcher" — daemon-first evolution | Preserves proven analysis engine, changes runtime model and surfaces |
| Knowledge scope | Layered: global (you) + project | Matches reality — one person, consistent preferences, different projects |
| Agent target | Claude Code primary, agent-agnostic architecture | Build for what you use, but design internals so adding other agents isn't a rewrite |
| Collaboration model | Solo user, project layer shared via git | CLAUDE.md in git is already team-shared; no team infrastructure needed |
| Storage model | Graph-modeled in SQLite | Think in graphs, store in tables. SQLite stays (proven, zero-dependency), schema evolves |
| Runtime model | Background daemon | Enables "suggestions ready before your next session" — work happens between sessions |
| Surfaces | TUI dashboard + in-session briefing | Two touchpoints: awareness inside sessions, full picture outside |
| Control model | Configurable: auto-apply or approval-first | Default to one mode, let user dial it per trust level |
| Ambition | Major version (Retro 2.0) | Get the architecture right, not incremental patches |

## System Architecture

Five layers, data flows upward, control flows downward:

```
┌─────────────────────────────────────────────┐
│              Surfaces                        │
│  (TUI Dashboard, In-Session Hook, CLI)       │
├─────────────────────────────────────────────┤
│              Projectors                      │
│  (Claude Code, Gemini, Cursor — pluggable)   │
├─────────────────────────────────────────────┤
│           Knowledge Store                    │
│  (Graph-modeled in SQLite: nodes + edges)    │
├─────────────────────────────────────────────┤
│             Analyzers                        │
│  (Pattern discovery, merge, confidence)      │
├─────────────────────────────────────────────┤
│             Observers                        │
│  (Session watcher, git events, file changes) │
├─────────────────────────────────────────────┤
│              Daemon (retrod)                 │
│  (Process lifecycle, scheduling, events)     │
└─────────────────────────────────────────────┘
```

## The Daemon (`retrod`)

A long-running background process that replaces the post-commit hook model.

### Process Management

- Started via `retro start`, stopped via `retro stop`
- Managed by launchd on macOS, systemd on Linux (installed by `retro init`)
- PID file at `~/.retro/retrod.pid` (existing lockfile mechanism evolves into this)
- Crash recovery: launchd/systemd auto-restart with backoff
- Logs to `~/.retro/retrod.log`

### Event Loop

- Polls observers on a configurable interval (default 60 seconds)
- Session ingestion is immediate (cheap, no AI)
- Analysis triggers when enough unanalyzed sessions accumulate (configurable threshold, default 3 sessions)
- Projection triggers when new knowledge is produced
- Auto-approved suggestions apply immediately; pending suggestions queue for review
- Only one AI call at a time (sequential)
- Optional active hours config to avoid burning tokens overnight

### Multi-Project Awareness

- Single daemon process watches all Claude Code sessions across all projects
- Project identity extracted from session file metadata (`cwd` → `git rev-parse --show-toplevel`)
- Each project tracked in the `projects` table with its agent type and last seen timestamp

## Knowledge Store

The brain. Graph-modeled in SQLite with two scopes.

### Node Types

| Type | What it represents | Example |
|------|--------------------|---------|
| `preference` | How you like things done | "Prefer short variable names" |
| `pattern` | Observed recurring behavior | "Always forgets to run tests before committing" |
| `rule` | An explicit directive | "Never mock the database in integration tests" |
| `skill` | A reusable capability | "TDD workflow for Rust" |
| `memory` | Factual context | "This project uses SQLite WAL mode" |
| `directive` | Strong instruction from the user | "Always use snake_case" |

### Edge Types

| Type | Meaning | Example |
|------|---------|---------|
| `supports` | Evidence for a node | session_observation → pattern |
| `contradicts` | Conflicts with another node | rule A ↔ rule B |
| `supersedes` | Replaces an older node | updated_rule → old_rule |
| `derived_from` | Generated from source | skill → pattern |
| `applies_to` | Scoped to a project | rule → project |

### Scopes

- **Global** — Your preferences, coding style, communication preferences with agents. Tagged with `scope = "global"`, `project_id = NULL`. Travels with you across all projects.
- **Project** — Conventions, architecture decisions, project-specific rules. Tagged with `scope = "project"`, `project_id = "<id>"`. Stays local to that project.

### Schema

```sql
nodes (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,       -- preference, pattern, rule, skill, memory, directive
    scope TEXT NOT NULL,      -- global, project
    project_id TEXT,          -- NULL for global, references projects.id
    content TEXT NOT NULL,
    confidence REAL NOT NULL, -- 0.0–1.0
    status TEXT NOT NULL,     -- active, pending_review, dismissed, archived
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
)

edges (
    source_id TEXT NOT NULL REFERENCES nodes(id),
    target_id TEXT NOT NULL REFERENCES nodes(id),
    type TEXT NOT NULL,       -- supports, contradicts, supersedes, derived_from, applies_to
    created_at TEXT NOT NULL,
    PRIMARY KEY (source_id, target_id, type)
)

projects (
    id TEXT PRIMARY KEY,
    path TEXT NOT NULL,
    remote_url TEXT,
    agent_type TEXT NOT NULL DEFAULT 'claude_code',
    last_seen TEXT NOT NULL
)
```

### Confidence Model

Preserved from v1:
- Single-session observations at 0.4–0.5
- Recurrence across sessions bumps confidence
- Explicit directives ("always"/"never"/"must") at 0.7–0.85 from a single session
- Confidence threshold (default 0.7) gates suggestion generation
- Global promotion threshold (default 0.85) suggests moving project knowledge to global

### Contradiction Detection

When new knowledge is added, the analyzer checks existing nodes for semantic overlap and flags potential contradictions as `contradicts` edges. Contradictions surface to the user: "You said X in project-foo but Y in project-bar — which is right?"

## Observers

The input layer. Detect events and feed raw data to analyzers.

### Session Observer (v2.0)

- Polls `~/.claude/projects/` on configurable interval (default 60 seconds)
- For each session file: has it been modified since last ingest? If yes, re-ingest the whole file
- Tags each session with its project (from `cwd` field, resolved via `git rev-parse --show-toplevel`)
- Queues for analysis when enough unanalyzed sessions accumulate
- Uses the existing ingest engine — proven parsing logic, no changes needed

### Git Observer (v2.1+, deferred)

- Watches for git events in known project directories
- Enriches session context: "this session resulted in these commits"
- Detects manual edits to CLAUDE.md or skills (user overrode Retro's output)

### File Observer (v2.1+, deferred)

- Watches agent context files (CLAUDE.md, MEMORY.md, skills/) for external changes
- Detects teammate PRs changing shared context
- Prevents Retro from suggesting something already added manually

### Observer Trait

```rust
trait Observer {
    fn poll(&mut self) -> Vec<Event>;
}

enum Event {
    SessionModified { path: PathBuf, project_id: String },
    GitCommit { project_id: String, hash: String },       // v2.1+
    FileChanged { path: PathBuf, project_id: String },     // v2.1+
}
```

## Analyzers

The existing analysis engine, evolved to produce knowledge graph operations.

### What Stays

- AI-powered pattern discovery via Claude CLI backend
- Pattern merging (AI-assisted + Levenshtein safety net)
- Confidence accumulation model
- Rolling window analysis across sessions
- Structured output via `--json-schema`
- `AnalysisBackend` trait and `ClaudeCliBackend` implementation

### What Changes

**Output target is the knowledge graph.** Instead of producing `Pattern` structs, the analyzer produces graph operations:
- `CreateNode { type, content, scope, confidence }`
- `CreateEdge { source, target, type }`
- `UpdateNode { id, confidence, content }`
- `MergeNodes { keep, remove }`

**Scope classification.** The analyzer decides global vs project for each piece of knowledge:
- References to specific code, architecture, or conventions → project
- Personal style, communication preferences, general coding habits → global
- Explicit "always"/"never" without project context → global
- Ambiguous → default to project, suggest promotion later

**Contradiction detection.** After producing new nodes, query existing nodes for semantic overlap and flag contradictions as `contradicts` edges.

**Cross-project context.** Because the daemon knows all projects, the analyzer can receive cross-project context to detect similar or conflicting knowledge across projects.

## Projectors

Agent-agnostic output layer. Translates knowledge into agent-specific files.

### Projector Trait

```rust
trait Projector {
    fn name(&self) -> &str;
    fn can_project(&self, node: &KnowledgeNode) -> bool;
    fn project(&self, nodes: &[KnowledgeNode], project: &Project) -> Vec<ProjectedFile>;
}

struct ProjectedFile {
    path: PathBuf,
    content: String,
    strategy: WriteStrategy,  // append, replace_section, full_rewrite
}
```

### Claude Code Projector (v2.0)

| Knowledge type | Output target | Method |
|---------------|---------------|--------|
| `rule`, `directive` | CLAUDE.md | Managed section or full management |
| `skill` | `.claude/skills/` | Two-phase generation (draft + validate) |
| `preference` | CLAUDE.md or skills | Depends on specificity |
| `pattern` | CLAUDE.md rule | "Avoid X" / "Prefer Y" style rules |
| `memory` | MEMORY.md | Read-only input, no writes |

### Shared vs Personal Output

The split follows **scope**, not output type:

| Output | Scope = Global | Scope = Project |
|--------|---------------|-----------------|
| Skills | Apply directly to `~/.claude/skills/` | PR to `.claude/skills/` in repo |
| Rules | Apply to `~/.claude/CLAUDE.md` | PR to project `CLAUDE.md` |
| Preferences | Apply directly | N/A |

Everything project-scoped that lives in the repo goes through a PR. Everything global that lives in the home directory applies directly.

### Future Projectors (not built in v2.0)

- Gemini projector → GEMINI.md / AGENTS.md
- Cursor projector → .cursorrules
- Generic markdown projector → any agent's context file

Adding a new projector means implementing the trait in `retro-projectors` — no changes to core.

## Surfaces

### In-Session Briefing

Retro maintains a per-project briefing file: `~/.retro/briefings/<project_id>.md`

The daemon regenerates this file when new knowledge is applied or suggestions are pending. Claude Code picks it up via existing context mechanisms (skill or CLAUDE.md reference).

**Content:**
- What was learned since last session
- What was auto-applied
- What's pending review

**Example:**
```markdown
## What's new since your last session

- **Applied:** "Always run cargo test before suggesting a commit" —
  added to CLAUDE.md (you made this correction 3 times last week)
- **Applied:** New skill "rust-error-handling" generated from your
  consistent patterns
- **Pending:** 1 suggestion needs your review — run `retro dash`
```

**Tone:** Concise, human, self-aware. Not a log dump.

### TUI Dashboard (`retro dash`)

Terminal UI for the full picture. Open when you want it, close when you're done.

**Panels:**

| Panel | Purpose |
|-------|---------|
| Activity feed | Timeline of daemon actions: sessions ingested, analysis runs, suggestions generated |
| Pending review | Suggestions waiting for approval. Preview, approve, or dismiss. Replaces `retro review` |
| Knowledge browser | Browse nodes by type and scope. See what Retro "knows" |
| Project overview | Known projects, agent type, last session, knowledge count |
| Daemon status | Running/stopped, last action, next scheduled, resource usage |

**Interaction:** Keyboard-driven (vim-style), approve/dismiss directly, quick filters by scope/type/status.

**Library:** `ratatui` + `crossterm`.

### CLI (preserved)

All existing commands still work. New commands:
- `retro start` / `retro stop` — daemon control
- `retro dash` — open TUI

Existing commands become secondary (manual triggers, debugging) rather than the primary workflow.

## Configuration & Control

### Config File (`~/.retro/config.toml`)

```toml
[daemon]
poll_interval_seconds = 60
analysis_trigger = "sessions"       # "sessions" or "time"
analysis_threshold = 3              # analyze after N new sessions
active_hours = "08:00-22:00"        # optional

[trust]
mode = "auto"                       # "auto" or "review"

[trust.auto_approve]
rules = true
skills = false
preferences = true
directives = true

[trust.scope]
global_changes = "review"
project_changes = "auto"

[knowledge]
confidence_threshold = 0.7
global_promotion_threshold = 0.85

[projects]
# Auto-discovered, overridable per-project
```

### Trust Modes

- **`auto`** — Most things auto-apply. Session briefing and TUI keep you informed. Best for experienced users.
- **`review`** — Nothing applies without approval. TUI shows pending suggestions. Best for new users or sensitive projects.

Fine-grained overrides available for power users who want something in between.

## Migration (v1 → v2)

### Database Migration (v3 → v4)

- `patterns` → `nodes` table. Type inferred from content (rules → `rule`, observations → `pattern`, directives → `directive`). Confidence carries over.
- `projections` → preserved for history, plus `derived_from` edges linking to source nodes.
- `sessions` → unchanged. `supports` edges created from session-to-pattern relationships.
- New `edges` and `projects` tables created.
- Migration runs automatically on first `retro start` or `retro init`.

### Hook Transition

- `retro init` on 2.0 removes old post-commit hook, installs launchd/systemd service
- Detects existing 1.x installation and offers migration prompt
- Old CLI commands still work if user declines migration

### Config Migration

- Old fields map to new structure, new fields get defaults
- Deprecated keys produce warnings, not errors

### File Layout

```
~/.retro/
├── config.toml          # evolved
├── retro.db             # schema v4
├── audit.jsonl          # unchanged
├── retrod.pid           # new (replaces lockfile)
├── retrod.log           # new (replaces hook-stderr.log)
├── briefings/           # new
│   └── <project_id>.md
└── backups/             # unchanged
```

### Principle

No data loss, no forced action. Upgrading without running `retro start` preserves the old workflow. The daemon is additive.

## Crate Architecture

### Current

```
crates/
├── retro-core/    # library (all logic)
└── retro-cli/     # binary (clap commands)
```

### Proposed

```
crates/
├── retro-core/        # library (knowledge store, analyzers, models, db)
├── retro-cli/         # binary (clap commands, manual triggers)
├── retro-daemon/      # binary (retrod — event loop, observers, scheduling)
├── retro-tui/         # library (ratatui dashboard)
└── retro-projectors/  # library (projector trait + implementations)
```

### Rationale

- **retro-core** stays the heart: knowledge store, analysis engine, models, DB, AI backend. No UI, no process management.
- **retro-daemon** owns the event loop, observer polling, scheduling, process lifecycle. Calls into retro-core for all logic.
- **retro-tui** is a library so both `retro dash` (CLI) and potentially the daemon can use it.
- **retro-projectors** separates agent-specific output from core. Adding a Gemini projector means adding a file here, not touching core.
- **retro-cli** slims down to a thin shell: parse commands, delegate to core/daemon/TUI.

### New Dependencies

| Crate | Purpose |
|-------|---------|
| `ratatui` | TUI rendering |
| `crossterm` | Terminal backend for ratatui |
| `notify` | Filesystem watching (daemon observer) |
| `signal-hook` | Graceful daemon shutdown |

### What Moves

- `retro-core/src/projection/` → `retro-projectors`, refactored behind `Projector` trait
- Everything else in retro-core stays
- CLI commands stay in retro-cli, some delegate to TUI components

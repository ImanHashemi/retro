# Retro 2.0 — "The Watcher"

A persistent brain that makes coding agents better automatically, session over session.

## Problem Statement

Retro 1.x works but doesn't get used. The five-command pipeline (ingest → analyze → apply → review → sync) requires the user to remember to run it. Even with a post-commit hook automating ingestion, the review step is manual, the terminal nudge is easy to miss, and across multiple projects it becomes a chore that gets skipped — defeating the entire purpose.

The core analysis engine is solid. What's broken is the runtime model (manual commands) and the surface (CLI output you have to go look at).

## Vision

- **Zero-effort improvement** — No commands to remember. The coding agent gets better session over session.
- **Noticeable difference** — Yesterday's mistakes don't happen today. The improvement is tangible.
- **Suggestions come to you** — Improvements surface themselves in-session and via a dashboard, ready to approve with minimal friction.
- **You stay in control** — Visibility into what's being learned and applied. Configurable trust level.
- **It's automatic** — The full loop from "observe mistake" to "suggest fix" to "easy approval" happens without user initiation.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture approach | "The Watcher" — scheduled job evolution | Preserves proven analysis engine, changes runtime model and surfaces |
| Knowledge scope | Layered: global (you) + project | Matches reality — one person, consistent preferences, different projects |
| Agent target | Claude Code primary, agent-agnostic architecture | Build for what you use, but design internals so adding other agents isn't a rewrite |
| Collaboration model | Solo user, project layer shared via git | CLAUDE.md in git is already team-shared; no team infrastructure needed |
| Storage model | Graph-modeled in SQLite | Think in graphs, store in tables. SQLite stays (proven, zero-dependency), schema evolves |
| Runtime model | Scheduled job (launchd/systemd) | Simpler than a custom daemon. Same "automatic, no commands" experience without process management complexity |
| Surfaces | TUI dashboard + in-session briefing | Two touchpoints: awareness inside sessions, full picture outside |
| Control model | Configurable: auto-apply or approval-first | Default to one mode, let user dial it per trust level |
| Default trust mode | `review` | Safe default for upgraders and new users. Users opt into `auto` when they trust the system |
| Ambition | Major version (Retro 2.0) | Get the architecture right, not incremental patches |

## System Architecture

Five layers, data flows upward, control flows downward:

```
┌─────────────────────────────────────────────┐
│              Surfaces                        │
│  (TUI Dashboard, In-Session Briefing, CLI)   │
├─────────────────────────────────────────────┤
│              Projectors                      │
│  (Claude Code, Gemini, Cursor — pluggable)   │
├─────────────────────────────────────────────┤
│           Knowledge Store                    │
│  (Graph-modeled in SQLite: nodes + edges)    │
├─────────────────────────────────────────────┤
│             Analyzers                        │
│  (Pattern discovery, merge, confidence)      │
├─────────────────────────────────────────────┤
│             Observers                        │
│  (Session watcher, git events, file changes) │
├─────────────────────────────────────────────┤
│          Scheduled Runner                    │
│  (launchd/systemd periodic job)              │
└─────────────────────────────────────────────┘
```

## Scheduled Runner

Replaces the post-commit hook model with a system-managed periodic job. No custom daemon process — launchd (macOS) or systemd (Linux) handles scheduling, lifecycle, and restart.

### How It Works

- `retro init` installs a launchd plist (macOS) or systemd timer (Linux) that runs `retro run` on a configurable interval (default 60 seconds)
- `retro run` is a new command that executes the full pipeline: observe → ingest → analyze → project → apply (for auto-approved items)
- Each invocation is a short-lived process: starts, does its work, exits. No long-running daemon, no PID file management, no signal handling.
- Existing lockfile mechanism prevents concurrent runs (two `retro run` invocations never overlap)

### `retro run` Pipeline

```
retro run
  ├── Acquire lock (exit if already running)
  ├── Poll session observer (check for modified session files)
  ├── Ingest new/modified sessions
  ├── Check analysis trigger (enough new sessions?)
  │   ├── Yes → run analysis → produce knowledge graph operations
  │   └── No → skip
  ├── Check for unprojected knowledge above threshold
  │   ├── Yes → run projectors → generate suggestions
  │   └── No → skip
  ├── Apply auto-approved suggestions (based on trust config)
  ├── Regenerate session briefings for affected projects
  ├── Log all actions to audit log
  └── Release lock
```

### Control

- `retro start` — loads/enables the launchd plist or systemd timer. If the plist/timer file doesn't exist, creates it first.
- `retro stop` — unloads the scheduled job but preserves the plist/timer file on disk. `retro start` re-loads it.
- `retro run` — single manual invocation of the full pipeline (useful for testing/debugging)
- Logs to `~/.retro/runner.log` (append, rotated at 1 MB, keeps 1 rotated file: `runner.log.1`)

### Cost Control

- `max_ai_calls_per_day` config (default 10) — hard cap on AI calls across all `retro run` invocations per day. Tracked in the `metadata` table.
- Active hours config (optional) — only run analysis during specified hours (e.g., `08:00-22:00`). Ingest still runs always (it's free).
- Analysis trigger threshold (default 3 sessions) — don't call the AI until there's enough new signal to justify it.

## Knowledge Store

The brain. Graph-modeled in SQLite with two scopes.

### Node Types

| Type | What it represents | Example |
|------|--------------------|---------|
| `preference` | How you like things done | "Prefer short variable names" |
| `pattern` | Observed recurring behavior | "Always forgets to run tests before committing" |
| `rule` | An explicit directive | "Never mock the database in integration tests" |
| `skill` | A reusable capability | "TDD workflow for Rust" |
| `memory` | Factual context about the project or user | "This project uses SQLite WAL mode" |
| `directive` | Strong instruction from the user | "Always use snake_case" |

**`memory` nodes** are context that informs analysis and projection but are not directly projected to any output file. They help the analyzer make better decisions (e.g., knowing the project uses SQLite helps it generate more relevant rules). They appear in the knowledge browser so the user can see what Retro knows, but they don't produce output artifacts.

### Edge Types

| Type | Meaning | Example |
|------|---------|---------|
| `supports` | Evidence for a node | session_observation → pattern |
| `contradicts` | Conflicts with another node | rule A ↔ rule B |
| `supersedes` | Replaces an older node | updated_rule → old_rule |
| `derived_from` | Generated from source | skill → pattern |
| `applies_to` | Scoped to a project | rule → project |

**`supersedes` semantics:** When node A supersedes node B, node B is automatically archived (`status = "archived"`). Archived nodes are excluded from projection and analysis context but preserved for history. The superseding node inherits the confidence of the superseded node (or higher, if new evidence warrants it).

### Scopes

- **Global** — Your preferences, coding style, communication preferences with agents. Tagged with `scope = "global"`, `project_id = NULL`. Travels with you across all projects.
- **Project** — Conventions, architecture decisions, project-specific rules. Tagged with `scope = "project"`, `project_id = "<id>"`. Stays local to that project.

**Scope conflict resolution:** Project scope overrides global. If global says "use spaces" and project says "use tabs," the project rule wins for that project. The projector merges both scopes when generating output, with project-scoped nodes taking precedence.

### Schema

```sql
CREATE TABLE nodes (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,       -- preference, pattern, rule, skill, memory, directive
    scope TEXT NOT NULL,      -- global, project
    project_id TEXT,          -- NULL for global, references projects.id
    content TEXT NOT NULL,
    confidence REAL NOT NULL, -- 0.0–1.0
    status TEXT NOT NULL,     -- active, pending_review, dismissed, archived
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE INDEX idx_nodes_scope_project ON nodes(scope, project_id, status);
CREATE INDEX idx_nodes_type_status ON nodes(type, status);

CREATE TABLE edges (
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    type TEXT NOT NULL,       -- supports, contradicts, supersedes, derived_from, applies_to
    created_at TEXT NOT NULL,
    PRIMARY KEY (source_id, target_id, type)
);

CREATE INDEX idx_edges_target ON edges(target_id, type);
CREATE INDEX idx_edges_source ON edges(source_id, type);

CREATE TABLE projects (
    id TEXT PRIMARY KEY,      -- hash of canonical path (git remote URL if available, else repo root)
    path TEXT NOT NULL,       -- current filesystem path
    remote_url TEXT,          -- git remote URL (for identity stability across moves)
    agent_type TEXT NOT NULL DEFAULT 'claude_code',
    last_seen TEXT NOT NULL
);
```

**Project identity:** `projects.id` is a human-readable slug derived from the repo directory name (e.g., `my-rust-app`), deduplicated with a numeric suffix if needed (e.g., `my-rust-app-2`). The slug is stable once assigned. `projects.remote_url` is used for reconnection when a repo is moved: if a session's `cwd` doesn't match any known `path` but its git remote matches a known `remote_url`, the project's `path` is updated. Repos without remotes use path-based matching only and orphan their knowledge if moved (acceptable trade-off for local-only repos). The slug is also used in briefing file paths (`~/.retro/briefings/my-rust-app.md`), making them human-inspectable.

**Foreign keys:** `PRAGMA foreign_keys` remains OFF (consistent with v1). Referential integrity is enforced in application code, not the database. This avoids migration complications with existing data.

**Schema versioning:** Uses existing `PRAGMA user_version` mechanism. v3 → v4 migration.

### Confidence Model

Preserved from v1:
- Single-session observations at 0.4–0.5
- Recurrence across sessions bumps confidence
- Explicit directives ("always"/"never"/"must") at 0.7–0.85 from a single session
- Confidence threshold (default 0.7) gates suggestion generation
- Global promotion threshold (default 0.85) suggests moving project knowledge to global

### Contradiction Detection (v2.1+, designed but deferred)

**Mechanism:** After each analysis batch produces new nodes, a dedicated AI call compares the new nodes against a summary of existing active nodes in the same scope (and global scope if analyzing a project). The AI is prompted to identify semantic contradictions — cases where two pieces of knowledge suggest conflicting behavior, even if the wording is different.

**Cost:** One additional AI call per analysis batch (not per node). The summary of existing nodes is truncated to fit within a reasonable token budget. This adds ~1 AI call per analysis run.

**Output:** Contradictions are stored as `contradicts` edges. They surface in the TUI knowledge browser (highlighted) and in session briefings ("Retro detected a potential conflict — please review").

**Deferred because:** Adding an AI call per analysis run doubles the cost of each run. The v2.0 focus is getting the core pipeline working (observe → analyze → project → suggest). Contradiction detection is valuable but not essential for the first release. The `contradicts` edge type exists in the schema from day one so this can be enabled without migration.

**Not in scope:** Automated contradiction resolution. The user decides which node to keep/dismiss.

## Observers

The input layer. Detect changes and feed raw data to analyzers.

### Session Observer (v2.0)

- Polls `~/.claude/projects/` by scanning for modified session files (mtime comparison against last ingest timestamp per file)
- For each modified file: re-ingest the whole file (overwrite previous ingest for that session)
- Tags each session with its project (from `cwd` field, resolved via `git rev-parse --show-toplevel`)
- Queues for analysis when enough unanalyzed sessions accumulate
- Uses the existing ingest engine — proven parsing logic, no changes needed
- No filesystem watching library needed — simple mtime polling, like the current `glob`-based approach

### Git Observer (v2.1+, deferred)

- Watches for git events in known project directories
- Enriches session context: "this session resulted in these commits"
- Detects manual edits to CLAUDE.md or skills (user overrode Retro's output)

### File Observer (v2.1+, deferred)

- Watches agent context files (CLAUDE.md, MEMORY.md, skills/) for external changes
- Detects teammate PRs changing shared context
- Prevents Retro from suggesting something already added manually

### Observer Trait

```rust
trait Observer {
    fn poll(&mut self, db: &Connection) -> Vec<Event>;
}

enum Event {
    SessionModified { path: PathBuf, project_id: String },
    GitCommit { project_id: String, hash: String },       // v2.1+
    FileChanged { path: PathBuf, project_id: String },     // v2.1+
}
```

## Analyzers

The existing analysis engine, evolved to produce knowledge graph operations.

### What Stays

- AI-powered pattern discovery via Claude CLI backend
- Pattern merging (AI-assisted + Levenshtein safety net)
- Confidence accumulation model
- Rolling window analysis across sessions
- Structured output via `--json-schema`
- `AnalysisBackend` trait and `ClaudeCliBackend` implementation

### What Changes

**Output target is the knowledge graph.** Instead of producing `Pattern` structs, the analyzer produces graph operations:
- `CreateNode { type, content, scope, confidence }`
- `CreateEdge { source, target, type }`
- `UpdateNode { id, confidence, content }`
- `MergeNodes { keep, remove }`

The analysis response JSON schema evolves to include `scope` (global/project) and the full set of node types.

**Scope classification.** The analyzer decides global vs project for each piece of knowledge:
- References to specific code, architecture, or conventions → project
- Personal style, communication preferences, general coding habits → global
- Explicit "always"/"never" without project context → global
- Ambiguous → default to project, suggest promotion later

**Contradiction detection (v2.1+).** Deferred to v2.1. The `contradicts` edge type exists in the schema but is not populated in v2.0. See the Knowledge Store section for the planned mechanism.

**Cross-project context.** The analyzer receives a summary of global knowledge when analyzing project sessions, enabling detection of project-vs-global conflicts and global promotion candidates.

## Projectors

Agent-agnostic output layer. Translates knowledge into agent-specific files.

### Projector Trait

```rust
trait Projector {
    fn name(&self) -> &str;
    fn can_project(&self, node: &KnowledgeNode) -> bool;
    fn project(
        &self,
        nodes: &[KnowledgeNode],
        project: &Project,
        backend: &dyn AnalysisBackend,  // needed for skill generation AI calls
    ) -> Result<Vec<ProjectedFile>, CoreError>;
}

struct ProjectedFile {
    path: PathBuf,
    content: String,
    strategy: WriteStrategy,  // append, replace_section, full_rewrite
}
```

**Note:** The `Projector` trait takes an `AnalysisBackend` parameter because skill generation requires AI calls (two-phase: draft + validate). Rule and preference projection is pure (nodes in, text out) and does not use the backend. This keeps the trait honest about its dependencies rather than hiding the AI requirement.

### Claude Code Projector (v2.0)

| Knowledge type | Output target | Method |
|---------------|---------------|--------|
| `rule`, `directive` | CLAUDE.md | Managed section or full management |
| `skill` | `.claude/skills/` | Two-phase generation (draft + validate) via AI backend |
| `preference` | CLAUDE.md or skills | Depends on specificity |
| `pattern` | CLAUDE.md rule | "Avoid X" / "Prefer Y" style rules |
| `memory` | (none) | Context for analysis only, not projected |

### Shared vs Personal Output

The split follows **scope**, not output type:

| Output | Scope = Global | Scope = Project |
|--------|---------------|-----------------|
| Skills | Apply directly to `~/.claude/skills/` | PR to `.claude/skills/` in repo |
| Rules | Apply to `~/.claude/CLAUDE.md` | PR to project `CLAUDE.md` |
| Preferences | Apply directly | N/A |

Everything project-scoped that lives in the repo goes through a PR. Everything global that lives in the home directory applies directly.

### Future Projectors (not built in v2.0)

- Gemini projector → GEMINI.md / AGENTS.md
- Cursor projector → .cursorrules
- Generic markdown projector → any agent's context file

Adding a new projector means implementing the trait in `retro-projectors` — no changes to core.

## Surfaces

### In-Session Briefing

Retro maintains a per-project briefing file: `~/.retro/briefings/<project_id>.md`

The scheduled runner regenerates this file when new knowledge is applied or suggestions are pending.

**Delivery mechanism:** `retro init` creates a skill file at `.claude/skills/retro-briefing.md` in each project that contains an instruction to read the briefing file at session start:
```markdown
---
name: retro-briefing
description: Read Retro's session briefing at the start of each conversation
---
At the start of each conversation, read ~/.retro/briefings/<actual-project-id>.md
if it exists and briefly acknowledge any new learnings or pending suggestions.
```

`retro init` resolves `<project_id>` to the actual project identifier when creating this file. This skill is project-specific and managed by Retro. It gets included in project PRs alongside other Retro-managed content.

**Content:**
- What was learned since last session
- What was auto-applied
- What's pending review

**Example:**
```markdown
## What's new since your last session

- **Applied:** "Always run cargo test before suggesting a commit" —
  added to CLAUDE.md (you made this correction 3 times last week)
- **Applied:** New skill "rust-error-handling" generated from your
  consistent patterns
- **Pending:** 1 suggestion needs your review — run `retro dash`
```

**Tone:** Concise, human, self-aware. Not a log dump.

### CLI (preserved)

All existing commands still work. New commands:
- `retro start` / `retro stop` — enable/disable scheduled job
- `retro run` — single pipeline invocation (what the scheduled job runs)
- `retro dash` — open TUI

**Legacy `--auto` flag:** Deprecated with warnings in v2.0, removed in v2.1. `retro ingest --auto` behaves as before for users who decline migration during the v2.0 cycle. The `--auto` chaining (ingest → analyze → apply) is superseded by `retro run`. Deprecation warnings appear on each `--auto` invocation directing users to `retro start`.

Existing commands become secondary (manual triggers, debugging) rather than the primary workflow.

## Configuration & Control

### Config File (`~/.retro/config.toml`)

```toml
[runner]
interval_seconds = 60               # how often the scheduled job runs
analysis_trigger = "sessions"        # "sessions" (after N new) or "time" (every N hours)
analysis_threshold = 3               # analyze after N new sessions
active_hours = "08:00-22:00"         # optional: only analyze during these hours
max_ai_calls_per_day = 10            # hard cap on AI calls per day

[trust]
mode = "review"                      # "auto" or "review" (default: review)

[trust.auto_approve]                 # only applies when mode = "auto"
rules = true
skills = false
preferences = true
directives = true

[trust.scope]                        # only applies when mode = "auto"
global_changes = "review"            # always review global changes even in auto mode
project_changes = "auto"

[knowledge]
confidence_threshold = 0.7
global_promotion_threshold = 0.85

[projects]
# Auto-discovered from sessions, overridable per-project
# [projects."my-rust-app"]
# agent = "claude_code"
# enabled = true
```

### Trust Modes

- **`review`** (default) — Nothing applies without approval. TUI and session briefing show pending suggestions. Safe for new users and upgraders.
- **`auto`** — Most things auto-apply based on `[trust.auto_approve]` and `[trust.scope]` settings. Session briefing and TUI keep you informed. Users opt into this when they trust the system.

## Migration (v1 → v2)

### Database Migration (v3 → v4)

**Deterministic type mapping** (no AI calls during migration):

| v1 PatternType | v1 SuggestedTarget | v2 Node Type |
|---------------|-------------------|--------------|
| `RepetitiveInstruction` | `ClaudeMd` | `rule` |
| `RepetitiveInstruction` | `Skill` | `directive` |
| `RecurringMistake` | any | `pattern` |
| `WorkflowPattern` | `Skill` | `skill` |
| `WorkflowPattern` | `ClaudeMd` | `rule` |
| `StaleContext` | any | `memory` |
| `RedundantContext` | any | `memory` |
| (any with confidence ≥ 0.85 and "always"/"never" in content) | any | `directive` |

- All migrated nodes get `scope = "project"` (safe default). Users can promote to global via TUI.
- `projections` → preserved for history, plus `derived_from` edges linking skill nodes back to source pattern nodes.
- `sessions` → unchanged. `supports` edges created from session-to-pattern relationships.
- New `edges` and `projects` tables created.
- Migration runs automatically on first `retro init` (v2) or `retro run`.

### `retro init` State Machine (v2)

| State | Behavior |
|-------|----------|
| Fresh install (no `~/.retro/`) | Create DB (v4), create config, install AND enable scheduled job (equivalent to `retro init` + `retro start`) |
| v1 detected (schema v3) | Prompt: "Migrate to v2? (y/N)". Yes: migrate DB, remove post-commit hook, install scheduled job. No: leave everything as-is, v1 commands still work |
| v2 already installed | Verify scheduled job is installed, update if needed |
| `--uninstall` | Remove scheduled job, remove hooks (if any), preserve data |
| `--uninstall --purge` | Remove everything including `~/.retro/` |

### Config Migration

- Old `[hooks]` section fields (`ingest_cooldown_minutes`, `analyze_cooldown_minutes`, etc.) map to new `[runner]` equivalents
- New fields get defaults
- Deprecated keys produce warnings, not errors

### File Layout

```
~/.retro/
├── config.toml          # evolved
├── retro.db             # schema v4
├── audit.jsonl          # unchanged
├── runner.log           # new (replaces hook-stderr.log)
├── briefings/           # new
│   └── <project_id>.md
└── backups/             # unchanged
```

### Principle

No data loss, no forced action. Upgrading without running `retro start` preserves the old workflow. The scheduled job is additive.

## Crate Architecture

### Current

```
crates/
├── retro-core/    # library (all logic)
└── retro-cli/     # binary (clap commands)
```

### Proposed

```
crates/
├── retro-core/        # library (knowledge store, analyzers, models, db)
├── retro-cli/         # binary (clap commands, includes TUI and runner)
└── retro-projectors/  # library (projector trait + implementations)
```

### Changes from v1

- **retro-core** stays the heart: knowledge store, analysis engine, models, DB, AI backend. Gains the observer trait and session observer implementation. No UI, no process management.
- **retro-projectors** is a new crate: separates agent-specific output from core. Contains the `Projector` trait and Claude Code projector. Adding a Gemini projector later means adding a file here, not touching core. Depends on retro-core for types and `AnalysisBackend`.
- **retro-cli** grows slightly: gains `retro run` (pipeline orchestration), `retro dash` (TUI), `retro start`/`retro stop` (scheduled job management). TUI code lives here as a module (not a separate crate) since it has only one consumer.
- No separate daemon crate — `retro run` is a CLI command, not a long-running process.

### New Dependencies

| Crate | Purpose |
|-------|---------|
| `ratatui` | TUI rendering |
| `crossterm` | Terminal backend for ratatui |

**Removed from original proposal:** `notify` (not needed — polling via mtime), `signal-hook` (not needed — no daemon process).

### What Moves

- `retro-core/src/projection/` → `retro-projectors`, refactored behind `Projector` trait
- Everything else in retro-core stays
- CLI commands stay in retro-cli, gains TUI module and runner command

## Testing Strategy

### Unit Tests

- **Knowledge store:** CRUD operations on nodes and edges, scope filtering, contradiction edge queries, confidence updates, `supersedes` archival behavior. Uses in-memory SQLite.
- **Observers:** Session observer mtime detection with fixture session files. Mock filesystem state.
- **Analyzers:** Existing `MockBackend` tests evolve to verify graph operation output instead of `Pattern` structs. Scope classification tests with fixture prompts.
- **Projectors:** Claude Code projector tested with fixture knowledge nodes → verify output file content and paths. Scope-based routing (global vs project output paths).
- **Migration:** v3 → v4 schema migration tested with fixture v3 databases.

### Integration Tests

- **Full pipeline:** `retro run` with `MockBackend` — ingest fixtures → analyze → project → verify output files.
- **TUI:** Snapshot tests for rendering (ratatui supports this). No interactive testing needed in CI.
- **Trust modes:** Verify auto-approve vs review behavior for different trust configs.

### Scenario Tests

- Existing scenario tests evolve to test the `retro run` pipeline instead of individual commands.
- New scenarios: scheduled job install/uninstall, migration from v1, briefing generation, TUI review workflow.

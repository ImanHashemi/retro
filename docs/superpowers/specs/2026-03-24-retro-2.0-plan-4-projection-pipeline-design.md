# Retro 2.0 Plan 4: Projection Pipeline — Design Spec

## Overview

Wire projection (file writes, PR creation, PR sync) into the v2 pipeline so the loop from "discover pattern → suggest → approve → apply to files" is fully closed without manual commands.

Currently `retro run` stops at analysis and tells the user to run `retro apply`. The dash approves nodes in the DB but doesn't write files or create PRs. This plan closes both gaps.

**Depends on:** Plans 1–3 (Foundation, Pipeline, Surfaces) — all complete on `feature/retro-2.0-surfaces`

**Parent spec:** `docs/superpowers/specs/2026-03-19-retro-2.0-the-watcher-design.md`

## Scope

### In Scope

- **`retro dash` immediate projection** for personal/global changes (rules, directives, preferences → `~/.claude/CLAUDE.md`)
- **`retro run` projection step** — project approved nodes to files, create batched PRs for project-scoped changes
- **`retro run` sync step** — check open retro PR state, dismiss nodes from closed PRs
- **Briefing regeneration** after projection
- **README update** — reorganize commands into primary/advanced sections
- **DB schema change** — add `projected_at` and `pr_url` columns to `nodes` table

### Out of Scope (Deferred)

- `retro curate` integration with knowledge graph (curate receiving v2 nodes as context, marking nodes projected after curate PR merge)
- Skill generation from the dash (requires AI call — always deferred to `retro run`)
- Global skill projection (skills always go through PR for now)

## Data Model Changes

Add two columns to the `nodes` table:

```sql
ALTER TABLE nodes ADD COLUMN projected_at TEXT;  -- NULL = not yet projected
ALTER TABLE nodes ADD COLUMN pr_url TEXT;         -- PR URL if projected via PR, NULL for direct writes
```

Schema version bumps to v5. The migration function adds a `if current_version < 5` block that runs both ALTER TABLE statements. This is consistent with the existing version-gated migration pattern in `db.rs`.

**Projection state logic:**
- `status = Active AND projected_at IS NULL` → needs projection
- `status = Active AND projected_at IS NOT NULL AND pr_url IS NULL` → projected directly (personal/global)
- `status = Active AND projected_at IS NOT NULL AND pr_url IS NOT NULL` → projected via PR, awaiting merge
- `status = Dismissed` → rejected (either by user or closed PR)

## `retro dash` Immediate Projection

When the user presses `a` (approve) on a node in the Pending Review tab:

**If node is global scope (project_id = NULL) AND type is rule/directive/preference:**
1. Write to `~/.claude/CLAUDE.md` managed section (append rule to managed section, create section if missing)
2. Set `status = Active`, `projected_at = now`
3. Show "Applied to ~/.claude/CLAUDE.md" message

**If node is global scope AND type is skill:**
1. Set `status = Active`, `projected_at = NULL` (defer to `retro run` — skill generation needs AI)
2. Show "Approved — skill will be generated on next run"

**If node is project scope:**
1. Set `status = Active`, `projected_at = NULL` (defer to `retro run` — needs PR)
2. Show "Approved — will be projected on next run"

### Implementation

The dash event handler (`tui/event.rs`) needs a new helper function for immediate global projection. The existing functions in `retro-core/src/projection/claude_md.rs` are:
- `build_managed_section(rules: &[String]) -> String` — builds section content from rule strings
- `update_claude_md_content(existing: &str, rules: &[String]) -> String` — merges rules into existing file content
- `read_managed_section(content: &str) -> Option<Vec<String>>` — reads current rules from managed section

Create a new function `project_rule_to_global_claude_md(content: &str) -> Result<(), CoreError>` that:
1. Reads `~/.claude/CLAUDE.md` (or creates empty string if missing)
2. Calls `read_managed_section()` to get existing rules
3. Appends the new rule
4. Calls `update_claude_md_content()` to produce the updated file
5. Creates `~/.claude/` directory if needed
6. Writes the file

This function lives in `retro-core/src/projection/claude_md.rs` alongside the existing functions. The dash imports and calls it.

Note: this introduces file I/O into the TUI event loop. Writing `~/.claude/CLAUDE.md` (a small file on the local filesystem) is fast enough to be imperceptible. If it becomes a problem, the write can be deferred to after TUI exit.

## `retro run` Projection Step

Add Step 4 to the pipeline (current Step 4 "Summary" becomes Step 5):

```
Step 4/5: Projecting approved changes...
```

### Logic

1. Query nodes where `status = Active AND projected_at IS NULL`
2. Group by project_id (NULL = global)
3. For each group:

**Global nodes (project_id = NULL):**
- Rules/directives/preferences → write to `~/.claude/CLAUDE.md` managed section
- Skills → AI call: generate + validate → write to `~/.claude/skills/`
- Set `projected_at = now` on all projected nodes

**Project-scoped nodes (grouped by project_id):**
- Rules/directives/preferences → write to `{project_path}/CLAUDE.md` managed section
- Skills → AI call: generate + validate → write to `{project_path}/.claude/skills/`
- Create `retro/updates-{YYYYMMDD-HHMMSS}` branch from default branch
- Commit all changes for this project
- Push + `gh pr create --base {default_branch}`
- Store PR URL on all projected nodes: `pr_url = {url}`, `projected_at = now`

**Cost control:** Skill generation uses AI calls. These count against `max_ai_calls_per_day`. If budget is exhausted, skip skill generation (project rules still go through).

**Error handling:** If PR creation fails (no `gh`, no remote, auth error), log the error but leave `projected_at = NULL` with a `projection_error` note in the audit log. The node will be retried on the next run. If it fails 3 times, set `projected_at = now` to stop retrying, and include the error in the briefing. The user can manually run `retro apply` as fallback.

**Working directory:** When projecting project-scoped nodes from launchd (global mode), the projection step must set `cwd` to the project's path from the `projects` table. Git commands for branch/PR creation require being inside the git repo.

### Reuse and Architecture

The `retro-projectors` crate (created in Plan 1) has a `Projector` trait and `ClaudeCodeProjector`. However, the current `ClaudeCodeProjector` implementation is a shell that calls `project_rules_to_claude_md()` — it doesn't yet handle skill generation or PR creation. For Plan 4, we take a pragmatic approach:

**CLAUDE.md projection** uses the existing functions in `retro-core/src/projection/claude_md.rs`:
- `update_claude_md_content(existing: &str, rules: &[String]) -> String` — merges rules into managed section
- `read_managed_section(content: &str) -> Option<Vec<String>>` — reads current rules

**Skill generation** uses the existing function in `retro-core/src/projection/skill.rs`:
- `generate_with_retry(backend: &dyn AnalysisBackend, pattern: &Pattern, max_retries: usize)` — two-phase generate + validate

This function takes a v1 `&Pattern`. Create an adapter: `fn node_to_pattern(node: &KnowledgeNode) -> Pattern` that maps `node.content` → `pattern.description` + `pattern.suggested_content`, `node.confidence` → `pattern.confidence`, `node.node_type` → `pattern.pattern_type` + `pattern.suggested_target`. This is a thin shim — v2-native skill generation can replace it later.

**Git branch/PR creation** extracts the core logic from `commands/apply.rs::execute_shared_with_pr()` into a reusable helper:

```rust
/// Create a git branch, commit files, push, and create a PR.
/// Returns the PR URL on success.
pub fn create_retro_pr(
    project_path: &str,
    files: &[(PathBuf, String)],  // (path, content) pairs to commit
    commit_message: &str,
) -> Result<String, CoreError>
```

This helper encapsulates: detect default branch → `git fetch` → `git stash` → `git checkout -b retro/updates-*` → write files → `git add` → `git commit` → `git push` → `gh pr create` → `git checkout -` → `git stash pop`. Lives in `retro-core/src/git.rs` as a reusable function.

### Relationship to `retro-projectors` crate

The `ClaudeCodeProjector` in `retro-projectors` is not used by Plan 4 directly. Plan 4 calls the underlying functions. A future plan can refactor to route through the `Projector` trait once v2-native projection is mature. The trait and crate remain available for future projectors (Gemini, Cursor).

## `retro run` Sync Step

Add Step 5 (after projection, before summary):

```
Step 5/6: Syncing PR state...
```

### Logic

1. Query nodes where `pr_url IS NOT NULL`
2. If none → skip entirely (no `gh` calls, zero overhead)
3. Group by unique `pr_url`
4. For each PR URL:
   - `gh pr view {url} --json state`
   - If `state = "MERGED"` → leave nodes Active, clear `pr_url` (successfully applied)
   - If `state = "CLOSED"` → set nodes to `status = Dismissed`, clear `pr_url`
   - If `state = "OPEN"` → no change (still waiting)

### Error handling

If `gh` is not available or the API call fails, log a warning and skip. Don't block the pipeline for sync failures.

### Relationship to v1 sync

This is a new `sync_node_prs()` function, separate from the existing `retro sync` command which operates on v1 `projections.pr_url`. The CLI `retro sync` command continues to handle v1 projections. The new sync function is called only from `retro run`. A future cleanup can merge them.

## Briefing Regeneration

After the projection step, regenerate the briefing file for each affected project:

```rust
// For each project that had nodes projected:
let applied_descriptions: Vec<String> = projected_nodes.iter()
    .map(|n| format!("{}: {}", n.node_type, truncate(&n.content, 80)))
    .collect();
let pending_count = db::get_nodes_by_status(conn, &NodeStatus::PendingReview)?.len();
let content = briefing::generate_briefing(project_id, &applied_descriptions, &[], pending_count);
briefing::write_briefing(project_id, &content)?;
```

Also regenerate for global changes (using a "global" briefing file or the current project's briefing).

For per-project briefing, filter pending count by project_id (not all pending nodes across all projects).

## `KnowledgeNode` Model Changes

Add two `Option<String>` fields with `#[serde(default)]`:

```rust
pub struct KnowledgeNode {
    // ... existing fields ...
    #[serde(default)]
    pub projected_at: Option<String>,  // RFC 3339 timestamp
    #[serde(default)]
    pub pr_url: Option<String>,
}
```

All DB read functions that construct `KnowledgeNode` (`get_node`, `get_nodes_by_scope`, `get_nodes_by_status`, `get_nodes_by_status`) must be updated to include columns 9 and 10 in their SELECT queries and row mapping. This is mechanical but touches multiple functions.

## README Command Reorganization

Primary workflow (what most users need):

| Command | Description |
|---------|-------------|
| `retro init` | Set up everything: database, config, background watcher |
| `retro dash` | TUI dashboard: review suggestions, browse knowledge |
| `retro start` / `retro stop` | Control the background watcher |
| `retro run` | Manual pipeline trigger (what the watcher runs) |
| `retro status` | Show session counts and knowledge summary |

Advanced / power-user commands:

| Command | Description |
|---------|-------------|
| `retro ingest` | Manual session ingestion |
| `retro analyze` | Manual pattern analysis |
| `retro apply` | Manual content generation (v1 pipeline) |
| `retro review` | CLI-based review (alternative to dash) |
| `retro sync` | Manual PR state sync |
| `retro patterns` | List v1 patterns |
| `retro diff` | Preview pending changes |
| `retro clean` | Archive stale patterns |
| `retro audit` | AI-powered context review |
| `retro curate` | AI-powered full CLAUDE.md rewrite |
| `retro log` | View audit log |

## Testing Strategy

### Unit Tests
- DB: `get_unprojected_nodes`, `mark_node_projected`, `mark_node_projected_with_pr`, `get_nodes_with_pr`, `dismiss_nodes_for_pr`, `clear_node_pr`
- Schema v5 migration: verify ALTER TABLE adds columns, existing v4 data preserved, new nodes get NULL defaults
- `project_rule_to_global_claude_md()`: write to temp dir, verify managed section created/updated
- `node_to_pattern()` adapter: verify field mapping
- `create_retro_pr()`: difficult to unit test (git/gh side effects) — covered by integration tests
- `sync_node_prs()`: mock `gh pr view` output, verify node status transitions

### Integration Tests
- `retro run --dry-run` shows the new step count (6 steps instead of 4)
- Dash approve for global rule: verify `~/.claude/CLAUDE.md` updated (needs temp HOME)

### Scenario Tests
- Update existing `v2-pipeline-dry-run` scenario to expect new step count
- Add scenario: full projection cycle with `--dry-run` (verify projection step output)

## Curate Integration (Deferred)

Not implemented in this plan. When implemented:
- `retro curate` receives v2 knowledge nodes as context alongside v1 patterns
- After curate PR is merged, `retro run` sync marks all Active rule/directive nodes for that project as projected
- This prevents the automatic pipeline from re-projecting rules curate already incorporated
